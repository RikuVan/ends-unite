### INTERNAL APIS

Goal: agree on and implement the same abstractions, with the same behaviours, across the stack.

Case: `Events` and `Participants` for a meetup registration service.

Frontend: `Typescript`

Backend: `Kotlin`

---

### EVENT

| Variant     | Data                                |
| ----------- | ----------------------------------- |
| `Pending`   | EventDetails ({date, place...})     |
| `Open`      | EventDetails                        |
| `Full`      | EventDetails                        |
| `Closed`    | EventDetails                        |
| `Cancelled` | EventDetails                        |
| `Error`     | `Server`({code, message, status})   |
|             | `NotFound`({code, message, status}) |

Behaviours: `is`, `match`, `fold`, `map`, `flatMap`, `from`, `tap` & iterable helpers

---

### PARTICIPANT

| Variant      | Data                                            |
| ------------ | ----------------------------------------------- |
| `Pending`    | ParticipantDetails ({name, email, affiliation}) |
| `Registered` | ParticipantDetails                              |
| `Waiting`    | ParticipantDetails                              |
| `Cancelled`  | ParticipantDetails                              |
| `Error`      | `Server`({code, message, status})               |
|              | `NotFound`({code, message, status})             |

Behaviours: `is`, `match`, `fold`, `map`, `flatMap`, `from`, `tap`

### Some questions you might ask (with short answers provided)

1. Could we make `Event` more generic. Yes: <code>`sealed class<out A>`</code>
2. Wouldn't it be better to define your behaviors once and reuse them. Yes. See `Arrow` for Kotlin and `ts-fp` for typescript.
3. Are we coupling the `ends` together? Nope.
4. What if I am happy in my own `end`. Sure, live and prosper.
